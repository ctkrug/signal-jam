# Architecture — Signal Jam

## Overview

A static, backend-free daily puzzle: a Rust/WASM core owns all game
truth (puzzle generation, sweep scoring), and a small TypeScript shell
renders whatever the core reports and forwards player input. No
project-specific game logic lives in TypeScript.

```
crate/                      Rust core, compiled to wasm32-unknown-unknown
  src/
    rng.rs                  SplitMix64 PRNG seeded from an FNV-1a hash of a string
    puzzle.rs                Puzzle::generate(date) — deterministic signal + decoys
    engine.rs                Engine — sweep state machine (lock/decoy/exhaustion)
    lib.rs                   wasm-bindgen boundary: PuzzleSession (JSON in/out)

web/                         Vite/TS web shell
  src/
    date.ts                  getUtcDateString() — today's puzzle seed
    spectrum.ts               computeRowAmplitudes() — pure waterfall amplitude math
    waterfall.ts              Waterfall — owns the <canvas>, scroll + paint
    audio.ts                  SfxPlayer — WebAudio synth SFX, mute persistence
    main.ts                   bootstrap: wires wasm + DOM + input + rendering
    style.css                 design tokens + all styling (see docs/DESIGN.md)
    wasm/                     generated by `npm run build:wasm`, gitignored
  test/                       vitest + jsdom
```

## Data flow

1. `main.ts` computes today's UTC date (`date.ts`) and constructs a
   `PuzzleSession` (wasm). The Rust side seeds an `Rng` from the date
   string and calls `Puzzle::generate`, placing one hidden signal and
   five decoys across the `[0, 1]` frequency band, each at least
   `MIN_SPACING` apart (`puzzle.rs`).
2. `PuzzleSession::puzzleInfo()` returns the puzzle's public shape
   (sweep budget, tolerances, every emitter's frequency/duty-cycle/
   noise-floor) as a camelCase JSON string, parsed once by `main.ts`.
3. Player input (pointer drag or arrow keys on `#sweep-track`) is
   converted to a normalized frequency (`frequencyFromPosition`) and
   passed to `PuzzleSession::sweep(frequency)` on every move.
4. `Engine::sweep` (`engine.rs`) is the sole authority on what
   happened: it edge-detects entry into the signal's or a decoy's
   detection window, consumes the sweep budget on a decoy's *first*
   hit only, and locks permanently on the signal. It returns a
   `SweepEvent` (serialized as `{ kind, ... }` JSON) — `main.ts` only
   renders that event, never re-derives game state.
5. Independently of sweep input, a `requestAnimationFrame` loop keeps
   calling `Waterfall.pushRow()` every frame using the latest known
   cursor frequency, so the waterfall scrolls continuously. Row
   amplitudes come from `computeRowAmplitudes` (`spectrum.ts`): each
   emitter is only visible within `REVEAL_RADIUS` of the cursor,
   except the locked signal, which renders full-bright thereafter.
6. `SfxPlayer` (`audio.ts`) turns events into WebAudio-synthesized SFX
   (oscillators/noise, no audio files); mute persists to
   `localStorage` and the `AudioContext` is created lazily on first
   call.

## Why the split

Puzzle truth in Rust/WASM (not TypeScript) keeps the daily puzzle
deterministic and testable independent of rendering, and gives the
60fps sweep loop headroom without a JS logic bottleneck — see
`docs/VISION.md`'s key design decisions.

## Running & testing

```
# Rust core
cd crate
cargo fmt --check
cargo clippy --all-targets -- -D warnings
cargo test

# Web shell (builds the wasm core first)
cd web
npm run build:wasm   # regenerates web/src/wasm/, required before dev/build/test-in-browser
npm run dev          # local dev server
npm run build         # static build into web/dist/ (subpath-relative)
npm run preview       # serve the built dist/
npm run lint
npm test              # vitest
```

`web/src/wasm/` is generated and gitignored — always run
`npm run build:wasm` (or `npm run dev` / `npm run build`, which do it
automatically) after any change under `crate/`.
